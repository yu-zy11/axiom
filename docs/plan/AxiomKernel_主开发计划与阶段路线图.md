# AxiomKernel 主开发计划与阶段路线图

本文档定义 `AxiomKernel` 从当前“可编译骨架阶段”到“工业级完整几何引擎阶段”的总路线、阶段目标、关键里程碑、退出标准和长期演进方向。该文档作为整个项目的主计划文档，后续开发、测试、文档、发布都应以本计划为总参照。

## 1. 当前阶段判断

截至当前，项目已完成：

- 文档体系初步成型
- 模块划分、接口边界、错误码、测试策略已定义
- **Stage 0**：基础代码框架、`Kernel` 门面与主要服务骨架、示例与 smoke 测试
- **Stage 1**：全量工程可编译；`Result`/诊断/错误码接入主链路；多模块专项 `ctest` 与 IO/布尔/表示等工作流回归形成门禁

当前阶段可定义为：

`Stage 1 已达成，处于 Stage 1.5 / Stage 2 过渡：核心公共层增强 + 基础几何/拓扑深化`

**本阶段退出标准（与代码、测试对齐的最低门禁）**：

- 根目录 `cmake` 配置与完整构建通过，全量 `ctest` 绿
- `Kernel::io_can_import_format` / `io_can_export_format` 与 `IOService::import_auto` / `export_auto` / `detect_format` 的格式集合**一致**（或由 API 文档明确双口径并有回归断言）
- 高风险链路（BOOL/HEAL/IO）关键诊断可在 `DiagnosticService` 检索，且 JSON 导出可携带工作流阶段字段（`Issue.stage`）以便 CI 归档

## 2. 最终目标定义

最终目标不是“做一个能演示的 CAD 内核”，而是构建一个在以下维度上全面领先传统闭源内核和开源内核组合方案的新一代几何引擎：

- 更强的鲁棒性
- 更现代的架构分层
- 更强的可诊断性
- 更好的缓存与增量重算能力
- 更好的混合表示支持
- 更好的插件化和云端扩展能力
- 更一致的 API 与工程化能力

## 3. 与现有内核竞争的核心能力目标

相对 `CATIA/CGM`、`SOLIDWORKS/Parasolid`、`ACIS`、`OCCT`，最终要追求的核心差异：

### 3.1 比传统商业内核更强的点

- 更细粒度的诊断和错误证据
- 更清晰的几何/拓扑/修复/缓存分层
- 更好的增量更新架构
- 更自然的混合表示支持
- 更好的插件和远程服务化能力

### 3.2 比开源内核更强的点

- 更强的布尔鲁棒性
- 更强的复杂圆角/抽壳能力
- 更成熟的工程流程和测试体系
- 更稳定的 API 和错误码体系

## 5. 当前到最终的详细任务树

## 5.1 第一层任务：把当前骨架变成稳固基线

必须完成：

- 编译通过
- 头文件依赖修正
- 门面接口和文档一致
- 基础测试通过
- 示例程序跑通

## 5.2 第二层任务：公共层与存储增强

必须完成：

- 统一 ID 分配器
- 统一诊断构造器
- 更清晰的记录结构
- 配置对象和全局策略管理

## 5.3 第三层任务：几何能力深化

必须完成：

- 解析几何求值增强
- `Bezier/BSpline/NURBS` 数据结构深化
- 参数域和最近点算法增强

## 5.4 第四层任务：拓扑能力深化

必须完成：

- 拓扑一致性检查增强
- 关系查询增强
- 写事务安全性增强

## 5.5 第五层任务：布尔核心

必须完成：

- 候选对生成
- 几何求交
- 切分
- 分类
- 重建
- 验证
- 可选修复

## 5.6 第六层任务：工业级特征

必须完成：

- 圆角倒角
- 偏置抽壳
- 替换面
- 删除面补面
- 直接编辑

## 5.7 第七层任务：工程化深化

必须完成：

- 更多测试数据集
- CI 完整闭环
- 性能基线
- 发布流程

## 6. 每一阶段的执行方式

每个阶段都按以下固定节奏推进：

1. 先修正文档与代码的不一致
2. 再构建最小可运行实现
3. 然后补测试
4. 再补诊断和错误码
5. 最后再扩功能和优化性能

## 7. 当前立即执行顺序

从当前阶段起，实际执行顺序建议如下：

### 第一步

- 跑通构建
- 修正编译问题
- 修正头文件依赖问题
- 修正门面与服务不一致问题

### 第二步

- 增强基础公共层
- 建立更稳的 `KernelState`
- 增强错误码与诊断构造方式

### 第三步

- 强化几何与拓扑最小能力
- 让更多基础对象真正可创建、可查询、可验证

### 第四步

- 推进基础建模和布尔第一代真实实现

## 8. 完成判定

“完成本几何引擎开发”不是单一节点，而应满足以下综合标准：

- 文档体系完整
- API 稳定
- 基础几何与拓扑稳定
- 工业级基础建模稳定
- 布尔、修复、导入导出主链路稳定
- 高级特征达到可用水平
- 性能和鲁棒性达到长期目标
- 测试、CI、发布和生态能力完整

## 9. 当前策略

从现在开始，按这个主计划持续推进时，优先级固定为：

1. 保证工程始终可构建
2. 保证文档与代码一致
3. 保证每一轮新增能力都有测试和诊断
4. 先把通用底座做稳，再做工业高级特征

## 10. 结论

这个项目如果要真正达到超越传统内核的目标，本质上是一个长期系统工程，而不是单一算法工程。当前最正确的路线不是直接堆功能，而是：

- 先稳住架构
- 再稳住基线
- 再推进核心建模能力
- 再逐步进入工业难点

从下一步开始，执行重点将进入：

`Stage 1: 编译基线与核心公共层稳定`
