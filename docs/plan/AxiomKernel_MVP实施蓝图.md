# AxiomKernel MVP实施蓝图

本文档定义 `AxiomKernel` 从立项到首个 `MVP` 版本的实施路径，目标是把“需求文档”和“技术架构文档”转化为可执行的研发计划、阶段目标、团队分工和交付标准。

## 1. 文档目标

本文档回答四个问题：

- `MVP` 具体做什么
- 先做哪些，后做哪些
- 团队如何分工
- 每个阶段如何验收

## 2. MVP定义

### 2.1 产品目标

`MVP` 的目标不是做一个完整 CAD 系统，而是做一个“可被上层 CAD/自动化建模系统调用的可用几何内核”。

### 2.2 MVP必须具备的能力

- 精确 `B-Rep` 基础表示
- 解析几何和基础 `NURBS`
- 顶点、边、面、壳、体的拓扑表达
- 基础构造体
- 拉伸、旋转、基础扫描
- 并、差、交三类布尔
- 基础查询和质量属性
- 基础验证器和基础修复器
- `STEP` 导入导出
- 三角化
- 事务、版本与统一诊断输出

### 2.3 MVP不纳入范围

- 高级角区圆角
- 变半径圆角
- 复杂抽壳和偏置
- 完整草图约束
- 完整历史树
- 云端协同
- 点云逆向
- 完整隐式建模

## 3. 研发周期建议

建议先按 `6 个月` 设计 `MVP`，如果团队较小可放宽到 `9 个月`。

推荐节奏：

1. `第 1 月`：基础设施和数据结构打底
2. `第 2 月`：几何求值和拓扑建模跑通
3. `第 3 月`：基础构造和查询能力落地
4. `第 4 月`：布尔与验证能力落地
5. `第 5 月`：数据交换、三角化、修复和性能打磨
6. `第 6 月`：稳定性收口、回归测试、示例集成和发布准备

## 4. 团队配置建议

### 4.1 最小团队

建议至少配置：

- `1` 名技术负责人
- `2` 名几何算法工程师
- `1` 名拓扑/内核基础设施工程师
- `1` 名数据交换与测试工程师
- `1` 名 QA/自动化测试工程师

### 4.2 理想团队

- `1` 名架构负责人
- `2` 名几何与鲁棒性工程师
- `2` 名布尔/修复/建模算法工程师
- `1` 名拓扑与事务工程师
- `1` 名数据交换工程师
- `1` 名性能工程师
- `1` 名测试工程师

## 5. 阶段拆分

## 5.1 阶段A：项目初始化与基础设施

### 时间

第 `1-2` 周

### 目标

- 建立代码仓库结构
- 搭建构建系统
- 搭建测试框架
- 建立错误码和诊断对象
- 建立基础数学模块

### 任务清单

- 创建模块目录结构
- 创建统一 `Result` 和 `Diagnostic` 类型
- 创建基础向量矩阵和 `BoundingBox`
- 确定句柄类型和版本 ID
- 创建单元测试模板
- 创建基线 CI

### 交付物

- 可编译的空内核工程
- 可运行的单元测试框架
- 基础类型库
- 代码规范说明

### 验收条件

- 工程可以跨平台编译
- 测试框架可自动执行
- 基础数学接口可用

## 5.2 阶段B：几何核心与拓扑核心

### 时间

第 `3-6` 周

### 目标

- 实现基础曲线曲面类型
- 实现拓扑结构和事务模型
- 实现几何求值接口

### 任务清单

- 实现直线、圆、平面、圆柱、球面
- 实现基础 `BSpline/NURBS` 描述结构
- 实现 `CurveFactory`、`SurfaceFactory`
- 实现 `CurveService`、`SurfaceService`
- 实现 `Vertex/Edge/Face/Shell/Body`
- 实现 `TopologyTransaction`
- 实现基础拓扑验证

### 交付物

- 可创建几何对象
- 可创建合法拓扑体
- 可运行基础求值和拓扑测试

### 验收条件

- 基础几何求值正确
- 能构造简单封闭体
- 非法拓扑可被识别

## 5.3 阶段C：基础建模能力

### 时间

第 `7-10` 周

### 目标

- 实现基础体构造
- 实现拉伸、旋转、基础扫描
- 实现基础查询与质量属性

### 任务清单

- 实现盒、球、柱、锥、环体
- 实现 `extrude`
- 实现 `revolve`
- 实现简单 `sweep`
- 实现截面
- 实现体积、面积、重心计算
- 实现点分类和最近点

### 交付物

- 基础构造模块
- 查询分析模块初版
- 示例脚本

### 验收条件

- 常见构造体可稳定生成
- 质量属性结果和参考值一致
- 截面和点分类结果稳定

## 5.4 阶段D：布尔与验证器

### 时间

第 `11-16` 周

### 目标

- 实现并、差、交
- 跑通面级候选过滤、求交、分类和重建
- 建立基础验证器

### 任务清单

- 实现 `AABB/BVH`
- 实现基础面面求交
- 实现边面切分
- 实现局部区域分类
- 实现基础 shell/body 重建
- 实现自交、拓扑和公差验证
- 实现错误类别和诊断报告输出

### 交付物

- `BooleanService` 初版
- `ValidationService` 初版
- 布尔案例集

### 验收条件

- 在基础零件库上并、差、交可用
- 失败可定位类别
- 不产生静默损坏模型

## 5.5 阶段E：修复、数据交换与三角化

### 时间

第 `17-20` 周

### 目标

- 支持 `STEP` 导入导出
- 支持三角化
- 支持基础自动修复

### 任务清单

- 实现 `STEP` 导入器
- 实现 `STEP` 导出器
- 实现 `BREP` 原生序列化
- 实现三角化
- 实现小边、小面、缝合修复器
- 将修复器接入导入流程

### 交付物

- 可互操作的格式模块
- 三角化模块
- 修复器模块

### 验收条件

- 可导入标准测试 `STEP`
- 导入后可运行验证
- 三角化结果可用于显示

## 5.6 阶段F：稳定性、性能和发布准备

### 时间

第 `21-24` 周

### 目标

- 建立性能基线
- 收口主要失败案例
- 形成外部可调用 SDK
- 发布 `MVP`

### 任务清单

- 建立黄金模型集
- 跑完整回归
- 修复高频布尔失败案例
- 优化三角化和查询缓存
- 编写示例工程
- 编写 API 文档
- 生成发布包

### 交付物

- `MVP` SDK
- 示例代码
- 性能报告
- 已知问题列表

### 验收条件

- 全量回归通过
- 关键性能指标达标
- 文档齐备
- 可由外部最小 demo 集成调用

## 6. 里程碑定义

### `M1` 工程基座完成

- 代码框架完整
- 数学基础可用
- 测试和 CI 打通

### `M2` 几何与拓扑可用

- 几何对象可创建求值
- 拓扑结构可构造和验证

### `M3` 基础建模可用

- 基础体、拉伸、旋转、查询可用

### `M4` 布尔初版可用

- 并、差、交可在标准样例集上跑通

### `M5` 数据交换与修复可用

- `STEP`、三角化、修复流程跑通

### `M6` MVP可发布

- 文档、测试、性能和 SDK 全部到位

## 8. 建议迭代节奏

建议采用 `2 周` 一个迭代。

每个迭代固定输出：

- 已完成能力
- 新增测试
- 性能变化
- 已知问题
- 下个迭代风险

## 9. 每阶段的验收指标

## 9.1 功能指标

- 是否覆盖阶段承诺的接口和能力
- 是否完成示例验证
- 是否能在标准数据集运行

## 9.2 正确性指标

- 单元测试通过率
- 回归测试通过率
- 质量属性误差是否在阈值内
- 布尔输出是否通过验证器

## 9.3 稳定性指标

- 不允许出现随机成功/随机失败
- 失败必须有稳定错误类别
- 不允许出现静默损坏输出

## 9.4 性能指标

- 几何求值有基线
- 典型布尔有基线
- 导入导出有基线
- 三角化有基线

## 10. 风险与应对

## 10.1 最大风险

- 布尔复杂度远超预估
- 容差系统设计不当导致全局返工
- 数据结构设计不适合后续版本化与缓存
- 导入脏模型场景过多拖慢 `MVP`

## 10.2 应对策略

- 先做标准样例上的稳定布尔，再扩复杂场景
- 容差系统第一月定原则，不后置
- 先把拓扑与事务边界做稳
- `MVP` 不追求导入所有脏模型自动修复

## 10.3 非建议做法

- 一开始同时做所有高级特征
- 把圆角、抽壳、布尔一起并行全面铺开
- 没有测试基线就做性能优化
- 没有失败案例归档就盲目修 bug

## 11. 输出物清单

`MVP` 完成时应至少有以下交付物：

- `AxiomKernel` 核心库
- 头文件或 SDK 包
- 示例程序
- 接口文档
- 架构文档
- 功能需求文档
- 测试报告
- 性能报告
- 已知问题列表
- 基础模型数据集

## 12. 与上层系统的最小集成方式

建议首版提供以下两种集成方式：

- `C++ SDK`
- `Python` 脚本绑定

首版最小示例应支持：

- 创建几何体
- 进行布尔操作
- 查询体积
- 导出 `STEP`
- 生成三角网格

## 13. MVP发布判定标准

若满足以下条件，可判定 `MVP` 可发布：

- `P0` 功能全部完成
- 核心回归测试通过
- 标准模型集上的布尔、导入导出和三角化稳定
- 有外部可跑通示例
- 文档完整
- 已知风险清单明确

## 14. MVP之后的下一步

`MVP` 发布后，建议按以下顺序继续演进：

1. 常规圆角与倒角增强
2. 偏置与抽壳增强
3. 增量缓存与局部重算增强
4. 插件化和稳定 ABI 强化
5. 混合表示和隐式体支持

## 15. 结论

`MVP` 的关键不是做得“多”，而是做得“稳”：

- 架构边界要稳
- 数据结构要稳
- 布尔和验证链路要稳
- 数据交换链路要稳
- 测试和诊断体系要稳

只要首版把这几个基础打牢，后续圆角、抽壳、直接编辑和混合建模都可以在这个底座上继续长出来。
